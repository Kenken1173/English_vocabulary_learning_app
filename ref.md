# オリジナル英単語学習アプリ 開発計画書

## 1. 開発目的

- **出題範囲の柔軟なコントロールをしたい！：** 塾の授業進度や宿題の範囲に合わせ、1ページ単位、1日単位での細かい出題範囲設定を実現する。
- **塾の方針に合わせた個別最適化を行いたい！：** 既存のアプリでは対応しきれない、塾独自の指導メソッドや生徒一人ひとりの習熟度に合わせた学習環境を提供する。

## 2. システム要件(現状考えていること)

- **プラットフォーム：** Webアプリ（PWA対応）
    - 中高生のスマホ利用を想定し、インストール不要でホーム画面から起動できる形式を採用。
- **技術スタック（予定）：**
    - フロントエンド：React
    - スタイリング：Tailwind CSS
    - バックエンド/DB：Supabase（開発スピードと運用コストを重視したBaaS構成）
- **MVP（最小限の機能）：**
    1. 単語登録機能（管理画面）：英単語・意味の登録。
    2. 単語テスト用紙作成画面（ユーザー画面）：登録された範囲から出題するA4テスト用紙をpdfで作成。

## 3. 塾長へのヒアリング事項（要件定義）

アプリの核となる「テスト形式」と「運用フロー」について、現場のニーズを確認する。

### ① テスト形式について

- **出題のアルゴリズムは？：**
    - 間違えた単語を優先的に出題する機能は必要か。
    - 合格基準（例：90%正解で合格）を設けるか。

### ② 学習コンテンツ（単語データ）について

- **データソース：**
    - 市販の単語帳（ターゲット、システム英単語など）に準拠した範囲設定か。
    - 塾独自のプリントや、教科書ごとのオリジナル単語リストを作成するか。
- **登録作業の運用：**
    - 単語の登録（入力）は誰が行うか（塾長、講師、または開発者側で一括登録か）
    - CSVなどで一括にアップロードするイメージか、一つ一つの単語を追加していくイメージか

### ③ 運用について

- **講師側の確認方法：**
    - 生徒の利用状況を一覧で確認できる管理画面は必要か。
- **利用デバイスの想定：**
    - 生徒個人のスマホ利用がメインと考えて良いか

---

## 4. 今後のロードマップ

1. **ヒアリングの実施：** 上記項目を塾長へ確認し、テスト形式を確定させる。
2. **画面設計（UIデザイン）：** 単語テスト画面と管理画面のラフ案を作成。
3. **DB設計：** ユーザー情報、単語データ、成績データの構造を定義。
4. **プロトタイプ開発：** MVP機能の実装を開始。
