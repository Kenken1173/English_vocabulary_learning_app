# 英単語学習アプリ 企画書

> **ステータス：** 要件定義中（ヒアリング未完了）
> **最終更新：** 2026-03-17

---

## 1. プロジェクト概要

塾の授業進度に合わせた英単語学習アプリを開発する。既存アプリでは実現できない「出題範囲の細かい制御」と「塾独自の運用フロー」への対応が最大の差別化ポイント。

### ターゲットユーザー
- **生徒（エンドユーザー）：** 中高生、スマホ利用がメイン
- **管理者：** 塾長・講師

---

## 2. 技術スタック

| レイヤー | 採用技術 | 理由 |
|---|---|---|
| フロントエンド | React | コンポーネント再利用性・エコシステム |
| スタイリング | Tailwind CSS | 開発速度 |
| バックエンド / DB | Supabase | BaaS構成で運用コスト・開発速度を両立 |
| 配信形式 | PWA | インストール不要でホーム画面起動を実現 |

---

## 3. MVP スコープ

以下の2機能のみをMVP（最初にリリースする最小限の機能）とする。

### 機能① 単語登録（管理画面）
- 英単語と日本語意味を登録・編集・削除できる
- 「教材名 / ページ / 日付」などの単位でグルーピングできる（詳細はヒアリング後に確定）

### 機能② テスト用紙作成（生徒向け画面）
- 登録された単語から範囲を指定して出題
- A4サイズのテスト用紙をPDF出力できる

### 意思決定メモ：Node.js バックエンド不採用について（2026-03-20）

**懸念：** 生徒のログイン情報・個人データを扱うにあたり、DBの接続情報がフロントエンドに露出することへのセキュリティリスク。

**判断：** Node.js バックエンドは不採用とし、Supabase のみの構成を維持する。

**理由：**
- Supabase の `anon` キーは公開前提の設計であり、露出しても問題ない
- データアクセス制御は **RLS（Row Level Security）** によりDBレベルで強制できる（例：生徒は自分のデータのみ参照可能）
- `service_role` キー（RLSを無視する管理者キー）はフロントエンドには置かない運用で担保
- Node.js で同等のアクセス制御を自前実装するより、RLS を正しく設定する方がセキュリティリスクが低い

**再検討トリガー：** CSVバルクインポート・外部API連携・複雑な認証フローが必要になった場合
